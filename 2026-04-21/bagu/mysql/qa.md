# MySQL 八股 Q&A 实战拷打

承接 [index.md](./index.md) 的索引基础，本文是 mock interview 模式下的高频追问 + 完整答案。三道大题层层递进：**索引选型 → 主键选型 → 分库分表 vs 分布式数据库**。

---

## Q1：MySQL 为什么用 B+ 树做索引？为什么不用哈希、红黑树、跳表、B 树？

### 🎯 设计目标先行

> 数据库索引的核心目标是 **「最小化磁盘 IO」**，因为内存和磁盘速度差 10 万倍，TB 级数据不可能全在内存。

### 🥊 逐个淘汰对手

| 数据结构 | 致命缺陷 | 适用场景 |
|---|---|---|
| **哈希表** | 不支持范围查询 / 排序 | InnoDB 自适应哈希索引（热点等值查询） |
| **红黑树** | 二叉树深度 log₂ n ≈ 20+，每层一次 IO | 内存场景（TreeMap） |
| **跳表** | 节点指针多，每页能塞 key 少，树高 | Redis ZSet（全内存） |
| **B 树** | 内部节点存 data 占空间；叶子无链表，范围查询要回溯 | 一些早期文件系统 |
| **B+ 树** ✅ | 完美 | **MySQL InnoDB / 大多数 OLTP 数据库** |

### B+ 树的 3 大杀手锏

1. **矮胖**：内部节点只存 key + 指针，一页 16KB 能塞 1000+ key，**三层树索引 10 亿行**
2. **数据全在叶子**：查询路径长度一致，延迟稳定（P99 可预测）
3. **叶子双向链表**：范围查询沿链表扫，IO 连续

### 数字估算

```
InnoDB 一页 = 16 KB
主键 BIGINT (8B) + 子节点指针 (6B) = 14B
一页能存 16384/14 ≈ 1170 个 key

3 层 B+ 树:
  根: 1170 指针
  中: 1170² ≈ 137 万指针
  叶: 137 万 × 16 行/页 ≈ 2200 万行 (一行 1KB)

→ 三层 B+ 树索引 ~2000 万 ~ 1 亿行
→ 四层 ~ 几百亿行
→ 这就是「单表 2000 万行」经验值的由来
```

### 🎤 一句话答

> 数据库要最小化磁盘 IO。**哈希不支持范围**；**红黑树二叉太深，IO 次数多**；**跳表节点指针多空间利用差**；**B 树内部存 data 占空间且范围查询要回溯**。**B+ 树矮胖（三层 10 亿行）+ 数据全在叶子（路径稳定）+ 叶子双向链表（范围扫高效）**，三大优势完美适配磁盘场景。Redis 用跳表 vs MySQL 用 B+ 树的本质差异是「内存 vs 磁盘」。

## Q2：订单表主键选 UUID / 自增 ID / Snowflake，怎么选？

### 🎯 需求拆解

```
日新增 100 万 → 年 3.6 亿 → 3 年 10 亿
峰值 2000 QPS
分布式（多服务实例同时写）
注定要分库分表
```

### 🥊 三方案对比

| | MySQL 自增 ID | UUID | Snowflake ✅ |
|---|---|---|---|
| 全局唯一 | ❌ 跨库重复 | ✅ | ✅ |
| 顺序 / 趋势递增 | ✅ 顺序 | ❌ 完全随机 | ✅ 趋势递增 |
| 页分裂 | ✅ 无 | ❌ 大量 | ✅ 几乎无 |
| 索引大小（每条） | 8B | 16B（占用 ×2） | 8B |
| 业务信息泄露 | ❌ 暴露日订单量 | ✅ | 部分（能反解时间） |
| 分布式生成 | ❌ DB 单点 | ✅ 本地生成 | ✅ 本地生成 |
| 性能 | 100% | 1/5 ~ 1/10 | ~95% |

### UUID 的 4 个致命伤

1. **页分裂崩塌**：随机插入触发大量 InnoDB 页分裂，写入 QPS 降到 1/5~1/10
2. **索引膨胀**：UUID 16B 比 BIGINT 8B 大一倍，**所有非聚簇索引叶子都存主键**，5 个二级索引 = 总索引膨胀 ~40 字节/行
3. **范围查询失效**：UUID 无序，按 ID 范围查时间没意义
4. **不可读不可排序**：日志、debug 极不友好

### Snowflake 64 bit 结构

```
| 1 bit 符号 | 41 bit 时间戳(ms) | 10 bit 机器 ID | 12 bit 序列号 |
| 0         | 69 年可用          | 1024 台机器     | 单机每 ms 4096 个 |
```

为什么完美：
- 高位时间戳 → 趋势递增 → B+ 树近似顺序插入
- 64 bit = 8 字节，和 BIGINT 一致，索引不膨胀
- 本地生成无 DB 依赖，百万 QPS 不是问题
- 不同机器 machineId 不同，天然分布式

### Snowflake 的 4 个生产坑

| 坑 | 解法 |
|---|---|
| **时钟回拨** | 检测后拒绝服务 / 等待时钟追上 / 借用未来时间戳 |
| **机器 ID 分配** | ZooKeeper / etcd 自动分配，避免手动配置冲突 |
| **跨机房** | 调整 bit 分配（牺牲序列号位换机房位） |
| **业务起始时间** | 自定义 epoch（如服务上线日），让 69 年从那天开始 |

### 业界变种
- **百度 UidGenerator**：解决时钟回拨
- **美团 Leaf**：号段 + Snowflake 双方案，生产成熟
- **滴滴 Tinyid**：基于号段

### 🎤 一句话答

> **选 Snowflake**。① 否决 UUID：随机主键导致 InnoDB 页分裂、索引膨胀一倍、范围查询失效；② 否决纯自增：分库后会全局重复，且暴露业务量被竞对算 GMV；③ Snowflake 高位时间戳保证趋势递增（无页分裂）+ 8 字节不膨胀 + 本地生成无 DB 单点。生产用美团 Leaf 等成熟方案，注意时钟回拨和机器 ID 分配两个坑。

## Q3：分库分表是什么？为什么不用 MySQL 主从/集群，或者直接用分布式数据库？

### 分库分表的 4 种操作

```
        一张大表 / 一个库容量到上限
                ↓
     ┌─────┬─────┬─────┬─────┐
     │     │     │     │     │
  垂直分库 垂直分表 水平分表 水平分库
```

| 类型 | 做法 | 解决什么 |
|---|---|---|
| **垂直分库** | 按业务模块拆库（user/order/product） | 业务隔离，团队边界清晰 |
| **垂直分表** | 按字段冷热拆表（user_basic + user_ext） | B+ 树更矮，cache 命中高 |
| **水平分库** | 同一张表数据散到多个库 | **解决容量和写吞吐**（核心） |
| **水平分表** | 同库内拆多张表 | 同上但粒度更细 |

实战通常 **水平分库 + 水平分表组合**：4 库 × 16 表 = 64 分片。

### 水平分片的 4 种策略

| 策略 | 优势 | 劣势 | 适用 |
|---|---|---|---|
| **Range** | 范围查询快、扩容容易 | 数据热点（最新分片热） | 日志、按时间归档的订单 |
| **Hash** | 分布均匀 | 范围查询要广播；扩容噩梦（rehash 全部） | 用户表、按 user_id 查 |
| **一致性 Hash** | 扩容只迁 1/N | 实现复杂 | KV 存储居多 |
| **Lookup（查询表）** | 最灵活 | 多一次查表 | 大客户专属分片 |

### 分片键（Sharding Key）选型

订单表常见做法：**分片键 = user_id 的 hash**，因为 C 端「我的订单列表」是高频查询。卖家「店铺订单」需求用**异构索引**（同样数据按 shop_id 再分一份）解决。

### 分库分表的 7 大痛点

| 痛点 | 解法 |
|---|---|
| 跨片 JOIN | 关联表用相同 sharding key |
| 跨片事务 | XA / TCC / Seata / 最终一致 |
| 聚合查询（COUNT） | 异构同步 ES、宽表预聚合 |
| 分页（深度分页） | 改成游标 `WHERE id > last_id` |
| 全局唯一 ID | Snowflake |
| 扩容迁移 | 翻倍扩容、双写、Canal 同步 |
| 跨片排序 | 业务限制只按分片键排 |

### 为什么不用 MySQL 主从？

```
[Master] ←── 所有写
   /  |  \
[S1][S2][S3] ← 读分流
```

- ✅ 读吞吐能扩
- ❌ **写吞吐不变**（master 单点）
- ❌ **容量没扩**（master 多大集群多大）

→ 100 万/天订单 3 年 10 亿，**master 装不下**，再多 slave 没用。

### 为什么不用 MySQL MGR / Cluster？

| 方案 | 局限 |
|---|---|
| **MGR (Group Replication)** | 多副本强一致，**还是同一份数据**，不解决容量；写性能还更差（共识开销） |
| **NDB Cluster** | 内存数据库 + SQL 不完全兼容，国内几乎没人用 |

→ MGR 是**高可用方案**，不是**容量扩展方案**。

### 中间件方案（大厂主流）

```
   [Sharding-JDBC / ShardingSphere / MyCAT]
       /     |     |     \
   [DB0] [DB1] [DB2] [DB3]
```

✅ 业务 SQL 几乎不改，中间件路由
✅ 用的还是 MySQL，运维成熟
❌ 复杂 SQL 支持有限

### 为什么不直接上分布式数据库（TiDB / OceanBase / CockroachDB）？

#### 优势（看起来很香）
- 业务无感（自动分片）
- 强一致（Raft / Paxos）
- 弹性扩容
- 完整 SQL 支持
- HTAP（OLTP + OLAP）

#### 4 大现实阻力

| 阻力 | 说明 |
|---|---|
| **历史包袱** | 阿里订单 20 年 MySQL 业务，迁移风险巨大 |
| **性能** | 单 SQL 比 MySQL 单机慢 2~5 倍（共识 + 跨节点） |
| **运维复杂** | TiDB 至少 8 节点起步，DBA 稀缺 |
| **生态** | ORM、Canal、监控、备份的兼容性都有坑 |

#### 适合分布式数据库的场景
- 单表 100 亿行起
- 强一致跨表事务多 → OceanBase
- 实时分析需求 → TiDB（HTAP）
- 跨机房多活 → OceanBase（金融级）

### 决策树

```
数据量?
├── < 1000 万              → 单 MySQL + 索引
├── 1000 万 ~ 10 亿
│    ├── 写少读多          → 主从 + Redis 缓存
│    └── 写多              → 分库分表 (Sharding-JDBC + Snowflake)
├── 10 亿 ~ 100 亿
│    ├── 标准 OLTP        → MySQL 分库分表
│    ├── 跨片事务多        → TiDB / OceanBase
│    └── 金融级 / 多活     → OceanBase
└── > 100 亿 / 海量分析    → HBase / ClickHouse / Doris
```

### 🎤 一句话答

> **分库分表**是按规则把大表数据散到多库多表，常用 hash 或 range 分片，分片键选 user_id 等高频查询字段。**主从只解决读不解决写和容量**；**MGR 是高可用方案不是容量方案**。**不直接上 TiDB/OceanBase** 是因为历史 MySQL 业务迁移风险、单 SQL 性能差 2-5 倍、运维复杂、生态成本高。**大厂主流是 MySQL + ShardingSphere + Snowflake + 异构索引**，新业务和金融场景才会上 OceanBase / TiDB——成熟生态 vs 新技术的实用主义选择。

---

## 📌 三道题串起来的总结

> **索引选 B+ 树是磁盘最优**；**主键选 Snowflake 是为了无页分裂 + 分布式唯一**；**分库分表是因为单 MySQL 容量上限，但不上分布式数据库是因为生态成本**。
>
> 三个决定背后都是同一个逻辑：**「在工程约束下做实用主义最优解」**——不追新技术，但用透成熟方案。
