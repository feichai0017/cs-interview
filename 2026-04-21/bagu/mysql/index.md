# MySQL 八股：索引原理（B+ 树）

**面试出现频率 No.1 的 MySQL 题**。能讲透就是会用 MySQL 的；讲不透就是只会写 SQL 的。

---

## 1. 为什么用 B+ 树？

这是面试官最喜欢「层层往下挖」的问题。要从**对手**说起。

### 候选数据结构对比

| 结构 | 查找 | 范围查询 | 磁盘 IO | 为什么不选 |
|---|---|---|---|---|
| **哈希表** | O(1) | ❌ 无序 | 多 | 不支持 `>`、`<`、`BETWEEN`、`ORDER BY` |
| **红黑树** | O(log n) | O(log n + k) | **太多** | 二叉树深度大，每查一次磁盘 IO 几十次 |
| **跳表** | O(log n) | O(log n + k) | 多 | 节点指针多，磁盘空间利用率差 |
| **B 树** | O(log n) | O(n)（中序遍历） | 中 | 内部节点也存数据，每页能放的 key 少；范围查询要回溯 |
| **B+ 树** ✅ | O(log n) | O(log n + k) | **少** | **专为磁盘 IO 优化** |

### B+ 树的三大杀手锏

#### ① 矮胖树（树高极低）
- 内部节点**只存索引（key + 子节点指针），不存数据**
- 一个 16KB 的页（MySQL InnoDB 默认页大小）能塞 1000+ 个 key
- 三层 B+ 树就能索引 10 亿条数据：`1000^3 = 10^9`
- **每次查询最多 3~4 次磁盘 IO**

```
              ┌──────────[40, 80, 120]──────────┐
             /                |                  \
       [10,20,30]      [50,60,70]          [90,100,110]
         |  |  |          |  |  |              |  |  |
         data叶子节点 (双向链表连接)
        ──────────────────────────────────►
```

#### ② 叶子节点双向链表
- 所有数据都在**叶子节点**
- 叶子节点之间用**双向指针**连起来
- 范围查询 `WHERE id BETWEEN 100 AND 500` 只要找到 100 的位置，**沿链表顺序读**，不用回到根重新查

#### ③ 内部节点紧凑
- 不存 data，每页能塞更多 key → 树更矮 → IO 更少

### 一句话总结
> **B+ 树 = 多叉（矮）+ 数据在叶子（内部紧凑）+ 叶子链表（范围快）**。这三个特性都是为了**最小化磁盘 IO**，而磁盘 IO 是数据库性能的命门。

## 2. 聚簇索引 vs 非聚簇索引（InnoDB 核心）

这是 MySQL 八股**最容易混淆的点**。两种说法实际上是不同维度的概念，但面试常常混着考。

### 定义

| | 聚簇索引（Clustered） | 非聚簇索引（Secondary / 辅助索引） |
|---|---|---|
| 叶子节点存什么 | **完整的行数据** | 主键值 |
| 一张表能有几个 | **只有 1 个** | 任意多个 |
| 谁是聚簇索引 | 主键；没主键则第一个非空唯一索引；都没有则隐藏的 `_rowid` | 除主键外的所有索引 |

### 物理存储理解

InnoDB 表本身就是一棵 B+ 树（按主键组织），**表 = 聚簇索引**。所以「数据」和「主键索引」物理上是同一份东西。

```
聚簇索引 (主键 id):
  叶子节点: [id=100, name='张三', age=20, addr='...', ...完整一行]

非聚簇索引 (name):
  叶子节点: [name='张三', id=100]   ← 只存主键值, 不存其他字段
```

### 回表（Back to Table）

查询 `SELECT * FROM user WHERE name = '张三'`：

```
1. 走 name 索引找到 name='张三' → 拿到 id=100
2. 拿着 id=100 回到聚簇索引再查一次 → 拿到完整行
   ↑↑↑ 这一步就是 "回表"
```

**回表 = 多走一棵 B+ 树**，多 3~4 次 IO。所以**回表是性能杀手**，能避免就避免。

## 3. 覆盖索引（Covering Index）

如果**查询的字段全部包含在索引里**，就不用回表，直接从索引拿数据返回。这就是覆盖索引。

```sql
-- name 列上有索引
CREATE INDEX idx_name ON user(name);

-- ❌ 需要回表 (要 age, 索引里没有)
SELECT name, age FROM user WHERE name = '张三';

-- ✅ 覆盖索引 (只要 name 和 id, 索引里都有, id 因为是主键自动带)
SELECT name, id FROM user WHERE name = '张三';

-- ✅ 联合索引 (name, age) 时, 这个查询就是覆盖索引
SELECT name, age FROM user WHERE name = '张三';
```

**EXPLAIN 看到 `Extra: Using index`** 就是覆盖索引在生效。

### 实战技巧
**避免 `SELECT *`**，只查需要的字段，配合联合索引，能让大量查询变成覆盖索引，吞吐量直接翻倍。

## 4. 联合索引 + 最左前缀原则

```sql
CREATE INDEX idx_a_b_c ON t(a, b, c);
```

这个索引能用在以下查询：

| 查询条件 | 能否走索引 | 走到第几列 |
|---|---|---|
| `WHERE a=1` | ✅ | a |
| `WHERE a=1 AND b=2` | ✅ | a, b |
| `WHERE a=1 AND b=2 AND c=3` | ✅ | a, b, c |
| `WHERE a=1 AND c=3` | ✅ | 只 a，c 用不上（中间断了） |
| `WHERE b=2` | ❌ | 完全不走索引（缺最左 a） |
| `WHERE b=2 AND c=3` | ❌ | 完全不走索引 |
| `WHERE a=1 AND b>2 AND c=3` | ✅ | a, b（b 是范围，c 用不上） |
| `WHERE a=1 ORDER BY b` | ✅ | a，且 b 排序免了 |

### 原因
联合索引在 B+ 树里的排列是**先按 a 排序，a 相同再按 b 排，再按 c 排**。

```
(a=1, b=1, c=5)
(a=1, b=2, c=3)  ← 同 a 内按 b 升序
(a=1, b=2, c=8)  ← 同 a,b 内按 c 升序
(a=2, b=1, c=1)
```

只有 `b=2` 没有 `a` 的话，B+ 树不知道从哪开始查（数据不是按 b 排的），只能全表扫。

### 重要例外：索引下推（ICP）
**MySQL 5.6+** 引入 **Index Condition Pushdown**：上面 `WHERE a=1 AND b>2 AND c=3` 的 c 条件虽然走不了索引，但**可以在索引层就过滤**，减少回表次数。EXPLAIN 看到 `Using index condition` 就是 ICP 在工作。

## 5. 索引失效的常见场景

每条都是面试经典考点：

### ① 对索引列做函数 / 运算
```sql
-- ❌ 索引失效
SELECT * FROM user WHERE YEAR(created_at) = 2026;
SELECT * FROM user WHERE id + 1 = 100;

-- ✅ 改写
SELECT * FROM user WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01';
SELECT * FROM user WHERE id = 99;
```
**原因**：B+ 树存的是原值，函数后的值不在树里，没法二分。**MySQL 8.0+ 支持函数索引** `CREATE INDEX ... ((YEAR(created_at)))` 可以救。

### ② 隐式类型转换
```sql
-- phone 是 VARCHAR 类型, 但传了数字
-- ❌ 失效, MySQL 内部对每行 phone 做 CAST(phone AS NUMERIC)
SELECT * FROM user WHERE phone = 13800001234;

-- ✅
SELECT * FROM user WHERE phone = '13800001234';
```
**反向**：字段是 INT，传字符串 `'100'` 不会失效，因为 MySQL 转字面量比转字段便宜。

### ③ `LIKE '%xxx'` 前缀模糊
```sql
-- ❌ 失效 (前面是%, 没法走 B+ 树)
WHERE name LIKE '%三'

-- ✅ 走索引
WHERE name LIKE '张%'
```
**为什么**：B+ 树是有序的，前缀确定才能定位。`%张` 等于不知道从哪开始。

### ④ `OR` 没有全部覆盖索引
```sql
-- name 有索引, age 没有
-- ❌ 整个查询不走索引 (因为 age 这部分必须全表扫)
WHERE name = '张三' OR age = 20

-- ✅ 改成 UNION
SELECT * FROM user WHERE name = '张三'
UNION
SELECT * FROM user WHERE age = 20
```

### ⑤ 范围条件后的列
联合索引 `(a, b, c)`，`WHERE a=1 AND b>5 AND c=3` 时 c 用不上（见上面联合索引表）。

### ⑥ `!=`、`NOT IN`、`IS NOT NULL`
通常会让优化器选择全表扫（认为返回行数太多），但**不绝对**——优化器会算 cost 决定。

### ⑦ 优化器认为全表扫更快
表很小、或者满足条件的行 > 全表 1/3 时，优化器主动放弃索引（**回表 IO 反而比全扫慢**）。`FORCE INDEX` 可以强制使用，但生产慎用。

## 6. 高频面试题速答

### Q1：B+ 树和 B 树的区别？
- B 树：**所有节点都存数据**，叶子不相连
- B+ 树：**只有叶子存数据**，叶子用双向链表连接
- B+ 树**树更矮**（内部节点能塞更多 key）+ **范围查询更快**（链表扫）+ **每次查询 IO 次数稳定**（必须查到叶子）

### Q2：为什么 InnoDB 主键要用自增 ID？
- B+ 树按主键有序排列，**自增 ID 永远在末尾插入**，不会触发**页分裂**
- 用 UUID 之类的随机主键，每次插入可能在中间，导致**大量页分裂**和**碎片**，性能急剧下降
- 自增 ID 还**短**（8 字节 vs UUID 36 字节），所有非聚簇索引叶子都存主键，主键短 → 整体索引小

### Q3：主键索引和唯一索引的区别？
- **数量**：主键 1 个，唯一索引可多个
- **NULL**：主键不允许 NULL，唯一索引允许（且多个 NULL 不冲突）
- **物理**：主键是聚簇索引，唯一索引是非聚簇

### Q4：为什么不用红黑树？
红黑树是**二叉树**，深度大。100 万行数据红黑树深度 ~20，每次查询 20 次磁盘 IO；B+ 树 3 层搞定，IO 次数差一个数量级。**红黑树是为内存设计的，B+ 树是为磁盘设计的**。

### Q5：为什么不用跳表？（MySQL vs Redis 对比题）
跳表是 Redis 的 ZSet 选择，因为 Redis 全内存，**节点空间利用率不重要**。MySQL 数据在磁盘，跳表节点的多层指针**浪费空间**（一页能塞的 key 少 → 树高 → IO 多）。

### Q6：索引越多越好？
**不是**。索引的代价：
- **空间**：每个索引都是一棵 B+ 树
- **写性能**：INSERT/UPDATE/DELETE 都要维护所有索引
- **优化器选错**：候选索引太多，优化器可能选个差的

经验：单表索引数 ≤ 5，单条查询用到的索引 ≤ 1（联合索引算 1 个）。

### Q7：什么是回表？怎么避免？
**回表**：通过非聚簇索引查到主键，再用主键查聚簇索引拿完整数据。
**避免**：
- **覆盖索引**：建联合索引覆盖查询字段
- **避免 `SELECT *`**：只查需要的字段

### Q8：索引下推（ICP）是什么？
MySQL 5.6+ 引入。原本范围条件后的列用不上索引，需要先回表再过滤；ICP 让 MySQL 在**索引层就用上这些条件过滤**，减少回表行数。

```
WHERE a = 1 AND b LIKE '%abc'    -- 联合索引 (a, b)
没 ICP: 走索引找 a=1 → 全部回表 → 过滤 b
有 ICP: 走索引找 a=1 → 在索引里就过滤 b → 满足的才回表
```

### Q9：count(*)、count(1)、count(字段) 区别？
- `count(*)`：MySQL 优化过的，**不会真的算 ***，直接走最小的索引（通常是主键）
- `count(1)`：等同于 `count(*)`，性能一致
- `count(字段)`：**会过滤 NULL**，且需要读出字段值，最慢

InnoDB 不像 MyISAM 缓存行数，`count(*)` 没有 `WHERE` 也得扫一遍索引。要快查总数：维护一个**计数表**或用近似值（`SHOW TABLE STATUS` 的 `Rows`，但不准）。

### Q10：什么是页分裂？
B+ 树叶子节点（页）有大小上限（16KB）。**插入时如果当前页满了，必须把页一分为二**，挪一半数据到新页。

- **顺序插入**（自增 ID）：永远在最后一页加，不会分裂
- **随机插入**（UUID）：可能在中间页插入，频繁分裂
- 页分裂会导致：**空间利用率降低**（每页只用一半）、**写入慢**、**碎片化**（OPTIMIZE TABLE 清理）

## 7. 实战诊断流程

工作中索引问题怎么排查？

```
1. 慢查询日志 (slow_query_log) 抓出慢 SQL
   ↓
2. EXPLAIN 看执行计划
   关注: type, key, rows, Extra
   ↓
3. type 是 ALL  → 全表扫, 加索引
   type 是 index → 全索引扫, 检查是否命中
   type 是 range/ref/eq_ref → 走索引
   ↓
4. Extra 是 Using filesort → ORDER BY 没走索引
   Extra 是 Using temporary → GROUP BY/UNION 用了临时表
   Extra 是 Using index → 覆盖索引 (好!)
   Extra 是 Using index condition → ICP (好!)
   ↓
5. SHOW INDEX FROM table 看现有索引, 分析是否冗余/缺失
```

### EXPLAIN 字段速记

| 字段 | 关注 | 好坏 |
|---|---|---|
| type | 访问方式 | const > eq_ref > ref > range > index > **ALL** |
| key | 实际用了哪个索引 | NULL = 没走索引 |
| rows | 估算扫描行数 | 越小越好 |
| Extra | 附加信息 | `Using index`/`Using index condition` 是好的；`Using filesort`/`Using temporary` 要警惕 |

---

## 8. 一句话总结

> **InnoDB 用 B+ 树是为了「矮 + 范围快 + IO 少」。表本身就是聚簇索引，主键索引存完整行，非聚簇索引存主键值，所以查非主键字段会回表。覆盖索引、最左前缀、避免索引失效是性能优化的三板斧；自增 ID 主键、避免 SELECT *、合理建联合索引是日常规范。**
